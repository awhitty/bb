**Bottom line — cleanest legal path:** generate the distillation dataset with a **locally-run Apache-2.0 Qwen3** teacher (Qwen3-32B or Qwen3-30B-A3B fits easily in the M5 Max's RAM), fine-tune Qwen3-0.6B/1.7B on it. Zero ToS exposure; the dataset and the resulting model are fully yours to redistribute, even commercially. Second-cleanest: distill from Claude under the explicit non-competing-model permission (this tool qualifies as a "specialized tool"). OpenAI is the least clear of the three.

## 2. Ranked technical recommendation

**Rank 1 — ship first (v0): grammar-constrained decoding + few-shot on a stock 0.6B–1.7B instruct model. No training.**
The bd query language is smaller than JSON — a GBNF grammar for it is maybe 30 lines. llama.cpp enforces grammars at the sampler level, so output is **syntactically valid by construction**; the model only chooses fields, operators, values, and dates. [node-llama-cpp](https://github.com/withcatai/node-llama-cpp) exposes exactly this (GBNF + JSON-schema grammars, "Enforce a JSON schema on the model output on the generation level") and **runs in Bun** (explicit support; a Bun-specific segfault was fixed in v3.17.0, Feb 2026). Few-shot prompt with ~10–20 NL→query examples + the field/enum list. A deterministic no-LLM parser alone (rank 3a) can't handle real NL ("stuff I touched last week that's still open"); this approach *is* the deterministic grammar, with a tiny LLM doing only the semantic slot-filling.

**Latency reality check (comfortably under 300ms):**
- Qwen3-0.6B decodes at **441–525 tok/s single-request on M4 Max** (MLX-based; [vllm-mlx paper, arXiv 2601.19139](https://arxiv.org/html/2601.19139v2)). A ~30-token query ≈ **60–70ms of decode**.
- M5 Max: 614 GB/s bandwidth, and GPU Neural Accelerators give **up to ~4× faster prefill vs M4** (MLX ≥0.30; Apple measured Qwen3-8B prefill 158→579 tok/s — [Apple ML Research](https://machinelearning.apple.com/research/exploring-llms-mlx-m5), [9to5Mac](https://9to5mac.com/2025/11/20/apple-shows-how-much-faster-the-m5-runs-local-llms-compared-to-the-m4/)). A 1–2k-token few-shot prompt prefills in well under 100ms on a ≤2B model.
- For scale: Gemma-2.3B does 158 tok/s and SmolLM3-3B 168 tok/s on M5 Max/MLX ([llmcheck.net benchmarks](https://llmcheck.net/benchmarks)) — so even a 4B model is borderline-fine (~200–300ms for 30 tokens), but ≤2B is the sweet spot. Keep the model resident; never cold-load per query.

**Rank 2 — the upgrade, and a genuine GO if the eval demands it: LoRA fine-tune Qwen3-0.6B (fallback 1.7B) with MLX-LM.**
Honest framing: for v0 this is over-engineering — but the cost is so low it's a legitimate week-two upgrade, and its *real* payoff is replacing the 1–2k-token few-shot prompt with a ~100-token prompt (prefill latency and prompt-maintenance both shrink), plus fixing systematic gaps (field hallucination, relative-date arithmetic) that few-shot leaves.
- **Evidence it works at this scale:** distil labs' text2sql — a tuned **4B matched a 685B teacher**, and a **0.6B hit 74%** (2× over base) ([distil-text2sql](https://github.com/distil-labs/distil-text2sql)); **Llama-3.2-1B >98% exact-match** on TinySQL ([arXiv 2503.12730](https://arxiv.org/pdf/2503.12730)); production NL→DSL SLM fine-tunes beating larger models on latency+quality ([arXiv 2604.09952](https://arxiv.org/pdf/2604.09952)); function-calling lineage — xLAM-1B at 78.94% BFCL beating GPT-3.5/Claude-3-Haiku ([Salesforce](https://huggingface.co/Salesforce/xLAM-1b-fc-r)), [Hammer](https://github.com/MadeAgents/Hammer)'s function-masking for on-device FC, and [Osmosis-Structure-0.6B](https://huggingface.co/osmosis-ai/Osmosis-Structure-0.6B) — literally a Qwen3-0.6B specialized for structured output.
- **Dataset size:** research says SLMs often need only **10–1,000 examples** to match larger models on narrow tasks; practitioner sweet spot 2,000–5,000 ([InsiderLLM](https://insiderllm.com/guides/fine-tuning-mac-lora-mlx/)); one documented success used [90 examples / 3 minutes](https://florinelchis.medium.com/fine-tuning-on-a-macbook-mlx-3-minutes-90-examples-and-a-model-that-actually-works-7de0547da347). **Target: ~1,000–3,000 synthetic pairs**, generated by prompting local Qwen3-32B with the grammar spec + seed examples + paraphrase/perturbation instructions (APIGen-style, per [xlam-function-calling-60k](https://huggingface.co/datasets/Salesforce/xlam-function-calling-60k)), validated by round-tripping every pair through the real bd parser and rejecting non-parses.
- **Training time on M5 Max: minutes.** A 3B LoRA took 3–4 min on an M2 Pro; 7B × 600 iters ≈ 12 min on M3 Max. A 0.6B on a few thousand examples is a **~5–15 minute run** — iteration is essentially free.
- **Eval:** hold out 15–20%; score (1) parse rate, (2) exact match, (3) *execution* equivalence against a fixture beads DB (same result set = correct, catches `p<=1` vs `p<2` equivalents). Keep the GBNF constraint at inference even after fine-tuning (the Osmosis pattern: tuned model + constrained decoding = belt and suspenders).

**Rank 3 — the honest alternatives:** (a) pure deterministic parser + embedding field-mapper: <10ms but only handles template-shaped input; worth keeping as a fast-path for inputs that already look like queries. (b) few-shot on a local 4–8B (Qwen3-4B/8B): works with zero effort but burns the latency budget and RAM for no accuracy benefit over constrained ≤2B on a grammar this tiny. Neither beats Rank 1.

## 3. Pragmatic v0 recommendation

1. **v0 (a weekend):** node-llama-cpp in-process in the Bun TUI + Qwen3-1.7B-Instruct GGUF (q4, ~1.1GB, resident) + hand-written GBNF for the bd query grammar + ~15 few-shot examples. Add a deterministic fast-path: if input already parses as a query, skip the model. Expected end-to-end: **<200ms**.
2. **Build the eval set regardless** (it's also the future training set): ~200 hand-checked NL→query pairs, scored by execution equivalence.
3. **Fine-tune only if the eval shows gaps** — then: `mlx_lm.lora` on Qwen3-0.6B, ~2k synthetic pairs from local Qwen3-32B, ~10-minute training runs, fuse adapter → convert to GGUF (`convert_hf_to_gguf.py`) → drop into the same node-llama-cpp slot with a stripped prompt.

**Tooling snapshot (mid-2026):** MLX-LM's `mlx_lm.lora` is the standard for mac fine-tuning (Unsloth remains CUDA-centric, axolotl not mac-native); wrappers if wanted: [mlx-tune](https://github.com/ARahim3/mlx-tune) (Unsloth-compatible API, SFT/DPO/GRPO) or [MLX-LoRA-Studio](https://github.com/Goekdeniz-Guelmez/MLX-LoRA-Studio) (native app). For serving inside Bun: **node-llama-cpp in-process is the right v0 call** (Bun support + generation-level grammars + Metal); the faster-but-heavier alternative is an MLX sidecar over localhost ([vllm-mlx](https://arxiv.org/html/2601.19139v2) is 21–87% faster than llama.cpp on Apple Silicon; [Rapid-MLX](https://github.com/raullenchai/Rapid-MLX) claims 0.08s cached TTFT) — only worth it if prefill dominates. Spawning `mlx_lm.generate` as a per-query subprocess is a non-starter (model load per call = seconds).

**Where evidence is thin (honestly):** no published tok/s benchmark specifically for Qwen3-0.6B *on M5 Max* (the 441–525 tok/s figures are M4 Max; M5 Max will be faster, mainly in prefill); the Alibaba Cloud API service-agreement text on output use was not publicly retrievable (moot given the local-weights path); Anthropic's "compete" line is a help-center gloss on the Terms, not the contract text itself — though it is Anthropic's own published interpretation.
